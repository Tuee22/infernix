# Build Artifacts

**Status**: Authoritative source
**Referenced by**: [../development/local_dev.md](../development/local_dev.md), [../../DEVELOPMENT_PLAN/phase-1-repository-and-control-plane-foundation.md](../../DEVELOPMENT_PLAN/phase-1-repository-and-control-plane-foundation.md)

> **Purpose**: Define where generated artifacts live and keep them out of tracked source paths.

## Build Roots

- the repo-local operator binaries live at `./.build/infernix` and `./.build/infernix-demo`; both
  share the `infernix-lib` Cabal library and ship in the same OCI image
- supported Apple host Cabal workflows call `cabal` directly with `--builddir=./.build/cabal` and
  `--installdir=./.build` and pass `exe:infernix exe:infernix-demo` so both binaries materialize
- supported outer-container Cabal workflows use `/opt/build/infernix`
- `cluster up` writes `./.build/infernix.kubeconfig` on the host path
- `cluster up` writes `./.build/infernix-demo-<mode>.dhall` on the host path
- `cluster up` writes `./.data/runtime/publication.json` as the publication inventory consumed by
  the demo workbench (when the demo surface is enabled), and host-bridge demo activation rewrites
  that same file when the demo `/api` moves to the Apple host bridge
- the web build stages generated PureScript contract modules under `web/src/Generated/`, written
  by `infernix internal generate-purs-contracts` (invoked by the `infernix-lib` build and again
  from `web/Dockerfile` so the web image build is self-contained)
- `spago bundle-app` produces the static demo bundle in `web/dist/`

## Generated Demo Config Publication

- `cluster up` stages `infernix-demo-<mode>.dhall` in the active build root
- the same content is then published into `ConfigMap/infernix-demo-config`
- in containerized execution contexts, the ConfigMap-backed file is mounted at `/opt/build/`
- the watched container path is `/opt/build/infernix-demo-<mode>.dhall`

## Rules

- repo-owned scripts or wrappers are not part of the supported build or launcher workflow
- generated demo-config files and kubeconfig output live under the active build root, not tracked
  source paths
- publication state lives under `./.data/runtime/` and is regenerated by `cluster up`,
  `cluster down`, or host-bridge demo activation
- generated PureScript contract modules stage under `web/src/Generated/` (rebuilt on every web
  build, not tracked in version control); the `spago bundle-app` output lives in `web/dist/`
- generated web build output and Playwright artifacts live under `web/dist/` and `./.data/` or
  the mounted equivalent, with atomic staging used so concurrent web builds do not expose
  partial `dist/` output
- engine-adapter Python builds use Poetry against `python/pyproject.toml`; outside the cluster,
  `poetry install --directory python` materializes `./.venv/` in the repo folder, and inside the
  engine container Poetry installs system-wide from the same `pyproject.toml`
- supported host Cabal usage passes `--builddir=./.build/cabal` and `--installdir=./.build`
  explicitly because the current Apple toolchain still ignores a project-file builddir default
- supported container Cabal usage always injects or enforces `/opt/build/infernix`
