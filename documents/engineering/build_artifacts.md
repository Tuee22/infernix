# Build Artifacts

**Status**: Authoritative source
**Referenced by**: [../development/local_dev.md](../development/local_dev.md), [../../DEVELOPMENT_PLAN/phase-1-repository-and-control-plane-foundation.md](../../DEVELOPMENT_PLAN/phase-1-repository-and-control-plane-foundation.md)

> **Purpose**: Define where generated artifacts live and keep them out of tracked source paths.

## Build Roots

- the repo-local operator binary lives at `./.build/infernix`
- supported Apple host Cabal workflows use `./cabalw ...`, which injects
  `--builddir=./.build/cabal` unless the caller already supplied a builddir
- supported outer-container Cabal workflows use `/opt/build/infernix`
- `cluster up` writes `./.build/infernix.kubeconfig` on the host path
- `cluster up` writes `./.build/infernix-demo-<mode>.dhall` on the host path
- `cluster up` writes `./.data/runtime/publication.json` as the publication inventory consumed by the routed workbench, and host-bridge service activation rewrites that same file when `/api` moves to the Apple host bridge
- the web build stages generated frontend contracts at `./.build/web-generated/Generated/contracts.js`
  on the host path and `/opt/build/infernix/web-generated/Generated/contracts.js` on the
  outer-container path

## Generated Demo Config Publication

- `cluster up` stages `infernix-demo-<mode>.dhall` in the active build root
- the same content is then published into `ConfigMap/infernix-demo-config`
- in containerized execution contexts, the ConfigMap-backed file is mounted at `/opt/build/`
- the watched container path is `/opt/build/infernix-demo-<mode>.dhall`

## Rules

- generated demo-config files and kubeconfig output live under the active build root, not tracked source paths
- publication state lives under `./.data/runtime/` and is regenerated by `cluster up`, `cluster down`, or host-bridge service activation
- generated frontend contracts stage under the active build root, and only the built runtime copy
  lives under `web/dist/generated/contracts.js`
- generated web build output and Playwright artifacts live under `web/dist/` and `./.data/` or the mounted equivalent, with atomic staging used so concurrent web builds do not expose partial `dist/` output
- unsupported bare host `cabal` invocations are not relied on for artifact isolation because the
  current Apple toolchain still ignores a project-file builddir default
- supported container Cabal usage always injects or enforces `/opt/build/infernix`
